## 2025-12-15

### Audit Whisper backend and UI boundary

- Goal: Identify where Whisper was used, how STT results flow into the UI, and which UI-facing glue must remain unchanged.
- Findings are listed below.
  - The UI uses Tauri commands/events only and was treated as out of scope for the refactor.
  - Dictation is push-to-talk (`Space` down starts, `Space` up finalizes).
  - The previous STT implementation used Whisper (`kalosm-sound`) with a pseudo-streaming loop over overlapping windows.
- Key files (before refactor): `src-tauri/src/session.rs`, `src-tauri/src/stt.rs`, `ui/src/App.tsx`.
- Outcome: The STT engine could be replaced while preserving the existing event/command contract.

### Add sherpa-onnx C API FFI crates

- Goal: Wrap the sherpa-onnx C API directly (bindgen + minimal safe wrapper) instead of relying on an unmaintained Rust crate.
- The following crates were added.
  - `crates/sherpa-onnx-sys`: build-time bindgen from `crates/sherpa-onnx-sys/vendor/sherpa_onnx_c_api.h`.
  - `crates/sherpa-onnx`: dynamic loading via `libloading`, safe RAII types (`OnlineRecognizer`, `OnlineStream`), JSON result parsing.
- Key decisions: Keep the bindgen surface minimal (online/streaming only) and avoid hard-linking native libraries.

### Implement sherpa-onnx streaming STT pipeline

- Goal: Replace Whisper pseudo-streaming with sherpa-onnx streaming while keeping the UI contract (`stt/partial`, `stt/final`) unchanged.
- Summary is listed below.
  - Decode work moved to a blocking worker; microphone capture stays in the audio callback thread.
  - Audio chunks are forwarded via channels; the worker produces partial and endpoint updates for event emission.
  - A “committed + live” transcript model avoids duplicate appends and commits on endpoints.
- Key files: `src-tauri/src/stt.rs`, `src-tauri/src/session.rs`, `src-tauri/src/audio/mic_stream.rs`.
- Runtime configuration: `AIR_SHERPA_ONNX_MODEL_DIR`, `AIR_SHERPA_ONNX_DYLIB` (documented in `spec/50_speech_to_text.md`).

- Removed: `kalosm-sound`, a Whisper-only benchmark binary, and Whisper-specific integration paths in the backend.
- Notes: Native sherpa-onnx runtime libraries and model files are required at runtime (with discovery/overrides).

## 2025-12-16

### Align streaming defaults with the Python mic demo

- Goal: Reduce behavior and accuracy differences between the Rust integration and the upstream sherpa-onnx Python microphone demo.
- Summary is listed below.
  - Matched endpoint rule3 default (`300.0` seconds).
  - Matched the default thread behavior (half of available CPU cores, minimum 1).
  - Batched microphone samples into ~100 ms chunks before `accept_waveform()` to match demo cadence and reduce drops.
- Notes: fp32 decoder remains the default; int8 decoder is an opt-in for speed.

### Compare the Python demo and align mic capture behavior

- Goal: Further reduce differences by aligning microphone capture and pre-processing with the reference demo.
- Summary is listed below.
  - Prefer 48 kHz capture when supported (fallback to device default).
  - Downmix multi-channel audio to mono (instead of taking only the first channel).
  - Clamp samples to `[-1.0, 1.0]` before feeding recognition.
  - Increase internal queue capacity to reduce drop risk under load.
- Notes: Remaining differences are most likely sustained load (callback cannot block) or device sample-rate/resampling differences.

### Two-pass dictation in the main backend (sherpa partials + whisper final)

- Goal: Use sherpa-onnx streaming for partials/endpoints and whisper-rs to produce final per-segment text.
- Summary is listed below.
  - On each sherpa endpoint, commit provisional text and queue the segment audio to a whisper worker thread.
  - When whisper completes, replace the previously committed segment and emit a new `stt/partial` update.
  - Finalization waits for outstanding STT work so `stt/final` reflects the whisper-corrected transcript.
- Docs updated: `spec/50_speech_to_text.md` (two-pass behavior and whisper env vars).

### Fix repeated segment insertion during silence (two-pass)

- Symptom: Silence could trigger repeated endpoint segments that produced repeated short phrases in the UI.
- Root cause: Silent endpoints were forwarded to whisper; whisper can hallucinate on near-silent audio.
- The fix is listed below.
  - Ignore sherpa endpoints with an empty transcript.
  - Add a simple silence gate in the whisper worker to skip second-pass on near-silent segments.

### Add reproducible STT trace and reduce “module fighting” symptoms

- Goal: Make dictation v2 issues reproducible and reduce visible text regressions and post-speech window hallucinations.
- Summary is listed below.
  - Added `AIR_STT_TRACE` / `AIR_STT_TRACE_DIR` to capture structured NDJSON traces and the recorded session WAV (`tmp/stt_trace/` by default).

### Fix overlay event listener churn and improve trace audio capture reliability

- Goal: Reduce the appearance of “modules fighting” by ensuring UI event handling is stable and ensure trace audio is still captured when sessions overlap.
- Summary is listed below.
  - Frontend: Register Tauri listeners once and gate events by the active session id stored in a ref, preventing accidental listener accumulation during session transitions.
  - Backend: Finalization writes the recorded session WAV whenever tracing is enabled, even if the user starts a new session before finalization completes, and emits a `finalize_detached` trace line when finalization is abandoned.
  - Backend: Suppress sherpa partial emissions until voice activity is observed in the current segment, reducing hallucinated text when holding Space in silence after an endpoint.
  - Backend: Fix trace WAV not being written by closing the update channel in the listening task, allowing finalization to receive `RecordedAudio`.
  - Backend: When committing a segment at an endpoint, fall back to sherpa text when the whisper window output has no overlap, preventing short hallucinated window commits from becoming segments.

### Fix leading-word regressions and segment overlap in dictation traces

- Goal: Address reproducible “extra leading word” failures observed in `tmp/stt_trace/` (for example, `So Sometimes ...` and `walking walking ...`).
- Root cause: An early endpoint could create an empty or low-quality first segment, and Whisper second pass could later fill it with a plausible but incorrect token. In addition, second-pass replacements could reintroduce overlap at segment boundaries.
- The fix is listed below.
  - Ignore endpoints with empty Sherpa text (treat as `EndpointReset`, do not create a segment).
  - Apply overlap de-duplication both when committing a segment and when applying Whisper second-pass replacements.
  - Collapse duplicated leading words in second-pass output when the repeated token is long enough to be a likely artifact.
  - Feed Sherpa 16 kHz audio to avoid internal resampler log spam.
- Added: `spec/55_stt_dictation_trace_regressions.md` and a small unit test set in `src-tauri/src/session.rs`.

### Switch macOS mic capture to Audio Queue (fix all-zero audio)

- Symptom: Microphone capture starts, but recorded audio is all zeros. Dictation shows no text and trace WAV files are silent.
- Root cause: Some AudioUnit-based capture paths can bind to the default output device. If the default output device is output-only (for example, an external USB DAC), the capture callback can read an input stream that does not exist, producing zeros.
- The fix is listed below.
  - Replace the macOS microphone backend with an AudioQueue (AudioToolbox) input queue that uses the default input device regardless of the default output device.
  - Update `research/stt_compare` microphone capture to use the same AudioQueue path and print `mic_stats mean_abs=... peak_abs=...` for quick diagnostics.
  - Document silent-trace troubleshooting in `spec/54_stt_dictation_pipeline_debugging.md`.
- Validation: `cargo make fmt`, `cargo make clippy`, and `cargo make nextest`.
